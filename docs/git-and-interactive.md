# Git 和交互式功能使用指南

## Git 仓库功能

### 概述
Git 功能允许你将加密的配置数据通过 git 进行版本控制和同步。这对于团队协作和配置备份非常有用。

### 前置要求
1. 数据路径必须是一个 git 仓库
2. 数据路径必须是 git 仓库的根目录（不能是子目录）
3. 如果需要推送到远程，需要配置远程仓库

### 可用命令

#### 1. 查看状态
```bash
senv git status
```
显示当前 git 仓库的状态，包括：
- 是否为 git 仓库
- 是否为根目录
- 当前分支
- 远程仓库地址
- 是否有待提交的更改

#### 2. 拉取更新
```bash
senv git pull
```
从远程仓库拉取更新。

**注意事项：**
- 如果有未提交的本地更改，pull 会失败
- 如果存在冲突，pull 会失败并提示手动解决
- 使用 `--ff-only` 策略，确保线性历史

#### 3. 提交更改
```bash
senv git commit -m "提交信息"
```
添加所有更改并创建提交。

**示例：**
```bash
senv git commit -m "更新数据库配置"
```

#### 4. 推送更改
```bash
senv git push
```
推送提交到远程仓库。

**注意事项：**
- 需要先配置远程仓库
- 如果没有需要推送的提交，会提示
- 如果是第一次推送，会自动设置上游分支

#### 5. 同步更改
```bash
senv git sync -m "提交信息"
```
一次性执行 add + commit + push。

**示例：**
```bash
# 使用自定义提交信息
senv git sync -m "更新生产环境配置"

# 使用自动生成的提交信息（时间戳）
senv git sync
```

### 使用场景

#### 场景 1：首次设置 git 同步
```bash
# 1. 初始化 git 仓库
cd ~/.config/senv/data
git init

# 2. 添加远程仓库
git remote add origin git@github.com:username/senv-configs.git

# 3. 创建初始提交
senv git commit -m "Initial commit"

# 4. 推送到远程
senv git push
```

#### 场景 2：日常更新
```bash
# 修改了一些环境变量或配置文件后
senv git sync -m "更新 API 密钥"
```

#### 场景 3：从远程拉取更新
```bash
# 在另一台机器上，拉取最新配置
senv git pull
```

### 错误处理

#### 错误：数据路径不是 git 仓库
```
❌ 数据路径 '/path/to/data' 不是 git 仓库。
请先初始化 git 仓库:
  cd /path/to/data
  git init
```

**解决方法：** 在数据路径下初始化 git 仓库

#### 错误：不是 git 根目录
```
❌ 数据路径 '/path/to/data' 不是 git 仓库的根目录。
请使用 git 仓库的根目录作为数据路径
```

**解决方法：** 使用 git 仓库的根目录作为 `--path` 参数

#### 错误：存在未提交的更改
```
❌ 无法 pull：存在未提交的更改。请先提交或暂存您的更改
```

**解决方法：** 先提交或暂存本地更改

#### 错误：pull 冲突
```
❌ pull 失败：本地和远程分支存在冲突。
请手动解决冲突后重试。
详细信息: ...
```

**解决方法：** 手动解决冲突
```bash
cd ~/.config/senv/data
git status  # 查看冲突文件
# 编辑冲突文件，解决冲突
git add .
git commit -m "解决冲突"
senv git push
```

## 交互式功能

### 概述
交互式模式提供一个用户友好的菜单驱动界面，让你可以轻松管理所有 senv 功能，无需记忆复杂的命令行参数。

### 启动交互式模式
```bash
senv interactive
```

### 主要功能

#### 1. 环境变量管理
- 查看所有环境变量
- 设置环境变量
- 删除环境变量
- 管理分组（创建、激活、停用）
- 导出环境变量

#### 2. 配置文件管理
- 查看所有配置文件
- 创建配置文件
- 编辑配置文件
- 导出配置文件
- 删除配置文件
- 查看配置详情

#### 3. Git 操作
- 拉取更新
- 提交更改
- 推送更改
- 同步更改（add + commit + push）
- 查看详细状态

#### 4. 会话管理
- 查看会话状态
- 清除会话

#### 5. 查看状态
- 查看系统整体状态

### 使用技巧

#### 快捷键
- `q`, `quit`, `exit` - 退出程序
- `h`, `help` - 显示帮助

#### 导航
- 输入菜单编号选择功能
- 输入 `0` 返回上级菜单

#### 默认值
- 很多提示都有默认值，直接按回车使用默认值
- 例如：分组名称默认为 `default`

### 最佳实践

1. **使用会话缓存**
   - 交互式模式会自动启动会话缓存
   - 避免每次操作都输入密码

2. **定期同步到 git**
   - 修改配置后及时提交
   - 使用有意义的提交信息
   - 定期推送到远程仓库

3. **分组管理**
   - 为不同环境创建不同的分组（dev, staging, production）
   - 只激活当前需要的分组

4. **配置文件**
   - 重要配置文件使用配置管理功能
   - 设置清晰的目标路径
   - 定期备份到 git

## 常见问题

### Q: 为什么要求数据路径必须是 git 根目录？
A: 这是为了避免意外提交其他文件。如果数据路径是 git 仓库的子目录，可能会提交父目录中的其他文件。

### Q: pull 失败怎么办？
A: 
1. 检查是否有未提交的本地更改
2. 如果有冲突，需要手动解决
3. 建议定期同步，减少冲突概率

### Q: 交互式模式和命令行模式有什么区别？
A: 
- **交互式模式**：适合探索和日常管理，界面友好，无需记忆命令
- **命令行模式**：适合脚本和自动化，支持管道和重定向

### Q: 可以在脚本中使用交互式模式吗？
A: 不建议。交互式模式需要用户输入，适合人工操作。脚本应使用命令行模式。

## 完整示例

### 示例 1：团队配置同步

**团队成员 A：**
```bash
# 1. 设置新配置
senv interactive
# 在交互界面中添加环境变量和配置文件

# 2. 同步到 git
senv git sync -m "添加新的 API 配置"
```

**团队成员 B：**
```bash
# 1. 拉取最新配置
senv git pull

# 2. 使用配置
senv env export | source
```

```bash

创建: production

选择 1 (环境变量管理) -> 2 (设置环境变量)

分组: staging
变量: DB_HOST=staging.db.example.com

分组: production

# 激活需要的分组

激活: development

# 提交更改

提交信息: 配置多环境数据库连接
```
